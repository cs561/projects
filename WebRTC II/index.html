Einführung WebRTC


Offenes Framework für das Web,
das Echtzeitkommunikation zwischen Browsern ermöglicht

beinhaltet grundlegende Strukturen

Netzwerk, Audio und Video Komponenten

Anwendung


Aufnahme, Kodierung und Peer-to-Peer Übertragung von Multimedia-Inhalten und Dateien

Chat, VoIP, Desktop-Sharing,  Document-Sharing, 
Video-Conferencing

Ziel


Entwickler sollen die Möglichkeit haben
Echtzeit Multimedia Applikationen
Plattformunabhängig 
nativ im Browser zu erstellen

Why WebRTC?


Offen, frei implementierbar, hohe Qualität
beinhaltet NAT und Firewall Traversal mit ICE
RTP-over-TCP und Proxy Unterstützung

Geschichte


Google kauft Global IP Solutions (Grundlegende Technik)
Mai 2011 -> Projekt “WebRTC” veröffentlicht von Google

IETF -> Standartisierung der Protokolle
W3C -> Browser API’s

W3C -> ConnectionAPI (work in progress)
Vorarbeit von WHATWG (Programmierschnittstelle)
fortgeschrittene Implementationen für Chrome, Opera und Firefox

1. Juni 2011 -> Google veröffentlicht Referenz Software Framework als freie Software (BSD-Lizenz)

Konkurrenz


WebRTC wird als Angriff auf das Monopol 
von Skype (Microsoft) gesehen

Microsoft arbeitet an einer Implementierung der Programmierschnittstelle für den Internet Explorer und will den eigenen Standard UC-RTC als Standard bei W3C etablieren

navigator.getMedia (

   // constraints
   {
      video: true,
      audio: true
   },

   // successCallback
   function(localMediaStream) {
      var video = document.querySelector('video');
      video.src = window.URL.createObjectURL(localMediaStream);
      video.onloadedmetadata = function(e) {
         // Do something with the video here.

   },

   function(err) {

Technik


Framework basiert auf HTML und JavaScript

Übertragung erfolgt mit (S)RTP
über die XMPP-Erweiterung Jingle
mit JavaScript Session Establishment Protocol (JSEP)

ausgehandelte Direktverbindung Peer-to-Peer

RTP over TCP


Multimedia-Streaming ohne Package-Loss

+ Nur 1 Port für Daten und Command
+ TCP Ports werden weniger gefiltert
+ TCP streamt Daten verlässlicher

- Kompliziertere Paketverarbeitung
- TCP Overhead kann zu delay führen

Browserunterstützung



ICE


IPv4-Adressen limitiert

Clients sind hinter NAT-Geräten (Router)

IP Adressen müssen gemapped werden

ICE Framework für sucht optimalen Verbindungsweg und baut die Verbindung auf (STUN + TURN)

STUN


Erkennt 
verschiedene NAT Typen, Firewalls
Traversal-Möglichkeiten 

Ermittelt
öffentliche IP-Adresse des Clients
Gegenstelle benötigt Adressierung für die Verbindung

TURN


Ermöglicht Client hinter NAT eine TCP/UDP Verbindung aufzubauen

Useful bei Symmetric NAT (kein STUN möglich)

Daten werden über Server geleitet
verursacht hohen Traffic
